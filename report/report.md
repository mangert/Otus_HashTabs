# Отчет по домашнему заданию: Реализация хеш-таблиц

## Общая информация

В проекте реализованы два варианта хеш-таблиц:

1. **[ChainHashTable](../headers/ChainHashTable.h)** — хеш-таблица с разрешением коллизий методом цепочек
2. **[OpenHashTable](../headers/OpenHashTable.h)** — хеш-таблица с открытой адресацией и квадратичным пробингом

Обе реализации наследуют абстрактный интерфейс **[IHashTable](../headers/IHashTable.h)** и поддерживают стандартные операции: вставка, удаление, поиск, доступ по ключу, рехэширование.

Для тестирования использовался специализированный класс **[HashTableTest](../headers/HashTableTest.h)**, который проводит серию функциональных и нагрузочных тестов с замером времени выполнения.

## Технические детали реализации

### Общие особенности
- Для хеширования используется `std::hash` (стандартная библиотека C++), что накладывает ограничение на тип ключа — он должен быть хешируемым (поддерживать `std::hash`)
- Поддержка копирования и перемещения (rule of five)

### Цепочечная хеш-таблица (ChainHashTable)
- В качестве контейнера для цепочек использован `std::list` (соответствует классическому определению метода цепочек)
- Каждый бакет представляет собой список пар ключ-значение
- Размер таблицы фиксируется при создании, при рехэшировании задается новый размер через параметры rehash

### Хеш-таблица с открытой адресацией (OpenHashTable)
- Квадратичный пробинг: `probe(hash, i) = (hash + A·i + B·i²) % M`
- Коэффициенты A и B настраиваются (по умолчанию A=0, B=1)
- Ленивое удаление (три состояния ячейки: EMPTY, ACTIVE, DELETED)
- Поддержка конфигурируемого максимального load factor (по умолчанию 0.75)
- Автоматическое рехэширование при достижении max load factor с коэффициентом роста φ = 1.618 (золотое сечение)

## Функциональное тестирование

### Пройденные тесты
- ✅ Граничные случаи (пустая таблица, один элемент, дубликаты)
- ✅ Удаление существующих и несуществующих элементов
- ✅ Очистка таблицы (clear)
- ✅ Копирование и перемещение (конструкторы и операторы присваивания)
- ✅ Само-присваивание
- ✅ Обработка исключений (размер = 0, некорректные коэффициенты)

### Тестирование исключений (OpenHashTable)
- Конструктор с M = 0 — выброшено `std::invalid_argument`
- Конструктор с A = 2, B = 2, M = 100 (gcd(2,100) ≠ 1) — выброшено `std::invalid_argument`
- Проверка, что A и B не могут быть одновременно равны нулю

## Выводы

1. **Производительность вставки**: 
   - На малых размерах (до 1 млн элементов) обе таблицы работают сопоставимо
   - На больших размерах открытая адресация показывает преимущество: вставка в **1.5-2 раза быстрее**

2. **Производительность удаления**:
   - Открытая адресация с ленивым удалением значительно эффективнее (в **4-5 раз** на больших размерах)
   - Метод цепочек требует больше времени на удаление из-за работы со списками

3. **Производительность рехэширования**:
   - Наибольший разрыв наблюдается при рехэшировании: открытая адресация быстрее в **5-12 раз**
   - Это объясняется более эффективным перемещением элементов (прямой доступ по индексу против обхода списков)

4. **Влияние коэффициентов пробинга**:
   - Различные комбинации A и B показывают близкую производительность
   - Критически важно соблюдение взаимной простоты ненулевых коэффициентов с размером таблицы

5. **Масштабируемость**:
   - Обе реализации демонстрируют линейную зависимость времени выполнения от размера данных
   - При росте с 1 млн до 100 млн элементов время увеличивается пропорционально

6. **Рекомендации по использованию**:
   - Для небольших таблиц (до 100 тыс. элементов) обе реализации пригодны
   - Для высоконагруженных систем с частыми удалениями предпочтительнее открытая адресация
   - Метод цепочек проще в реализации и отладке, но уступает в производительности

## Заключение

Обе реализации хеш-таблиц полностью соответствуют требованиям задания и успешно проходят все тесты. Открытая адресация с квадратичным пробингом и ленивым удалением показывает лучшую производительность на больших объемах данных, особенно при операциях удаления и рехэширования. Метод цепочек, использующий `std::list`, является классической реализацией и служит хорошей базой для сравнения.
